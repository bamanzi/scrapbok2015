    Everything you need to know about containers, minus the hype

    Containers are like virtual machines, only more lightweight and efficient. They create an OS within your host OS in which you can install and run apps. You could run a Debian host OS with multiple Fedora, Ubuntu or CentOS containers. You can run multiple versions of apps all in their containers. This decoupling of apps from the host OS delivers an extraordinary level of flexibility and benefits.

    Simple

    Backing up your app is as simple as zipping the container folder. Moving apps across servers is as simple as moving the zip file. You can freeze and distribute state with clones and snapshots. Since the container is a folder sharing data between host and container is easy. This make running apps especially on servers extremely convenient.

    Fast

    Containers operate at bare-metal speed and do not have the performance overhead of virtualization. This is because containers do not emulate a hardware layer and use cgroups and namespaces in the Linux kernel to create lightweight virtualized OS environments.

    Since you are not virtualizing storage a container doesn't care about underlying storage or file systems and simply operates wherever you put it.

    Containers are also easier to use, storage is just a folder, you do not need to dedicate resources upfront and they startup and shut down in seconds.

    This fundamentally changes the way we virtualize workloads and applications, as containers are simply faster, more portable and can scale more efficiently than hardware virtualization, with the exception of workloads that require an OS other than Linux or a specific Linux kernel version.

    Portable

    The biggest benefit of containers apart from performance is portability. You server is basically mobile and you are not stuck to any cloud, provider or server. They make your apps and data portable across servers.  Simply zip the container folder and move. This also makes basic management operations like backups, cloning, snapshots, deployment extremely simple.

    Because apps and workloads are isolated users can run multiple versions of PHP, Python, Ruby, Apache happily coexisting tucked away in their containers, and get cloud like flexibility of instances and workloads that can be easily moved across systems and cloned, backed up and deployed rapidly.

    Container technology is not new, FreeBSD has Jails, Solaris has Zones and there are other Linux container technologies like OpenVZ and Linux VServer. However they depend on custom kernels impeding adoption. Current Linux container technology is backed by mainstream support in the kernel paving the way for widespread adoption.

    The Genesis

    Containers owe their origin to the development of cgroups and namespace support in the Linux kernel. Namespaces allow you to run isolated processes. LXC is the user land part of the project to develop a container manager. Container managers like LXC, Docker, Nspawn and others combine namespaces with chroots to provide end user containers. Here is an overview on how Linux containers work.

    The LXC project offers a nice set of tools for container management, advanced networking and storage support for things like cloning and snapshots and a wide choice of minimal container OS templates.  So how do you use containers?

    Basics

    You create a container by specifying the container OS template and the name of the container. In this case we are choosing Ubuntu and calling the container P1. You can choose to use Debian, Fedora, CentOS or other OS templates.

    lxc-create -t ubuntu -n p1

    You start it with the lxc-start program

    lxc-start -n p1

    You can access the container by directly attaching to it, or over the network by ssh

    lxc-attach -n p1

    You can get information on running containers using

    lxc-ls -f

    You can poweroff or reboot the container from within the container or use lxc-stop from the host.

    To clone a container simple use the lxc-clone command. If you are using the Btrfs or LVM Thin Pools you will get efficient CoW clones.

    lxc-clone -o p1 -n p2

    You can also create snapshots of containers by using the -s flag if the system supports Overlayfs or LVM Thin Pools.

    lxc-clone -s -o p1 -n p2

    With Overlayfs snapshots you get 'linked snapshots'. The root filesystem of p1 is mounted as the read only root fs of p2 with changes going to a delta directory in the p2 container. With LVM Thin Volumes you get thin snapshots.

    To move the container simply zip it and move across servers and it will work as you left it. You can try LXC in seconds with the lightweight Flockbox VM based on Alpine Linux. It's a 80MB download and available for Virtualbox, VMWare and KVM.

    Container Networking

    Containers need to be on the network to be useful and accessible. Containers have a full stack networking layer in their own namespace. From basic bridging and NAT you get a wide range of capabilities from Vlans, Macvlans and physical devices.

    Container Storage

    You don't need to think about container storage since a container is just a folder on your file system. The OS, apps and data are in the folder. When combined with file systems like Btrfs, LVM or Overlayfs you get the advanced features of these file systems like efficient CoW clones, disk quotas and Overlayfs snapshots.

    Unprivileged Containers

    Unprivileged containers use user namespace support in the kernel to give you containers that can be run by non root users. This is a big step forward for container security and paves the way for multi-tenant workloads. LXC has supported unprivileged containers since their 1.0 release in 2013.

    Container Technology Choices

    The LXC project is now working on LXD which extends LXC capabilities with multi-host container management and soon live migration.

    There are now other container managers like Docker,  Systemd-nspawn, Firejail, Pflask among others.

    The main difference between LXC and Docker is LXC is more focused on OS containers that are like lightweight VMs and Docker on single app containers targeted towards a PAAS centric use case. OS containers run an init and thus support multiple processes, and App containers don't and can only run single apps.

    Docker and others are pushing a PAAS centric use case of containers. Our article on using Overlayfs and LXC provides an overview of how layered containers work.

    The Elephant in the Room

    Docker has marketed itself aggressively and has got conflated to container technology rather than its specific use case of containers. This sort of led to less informed discussion and widespread misconception.

    Docker is a very specific PAAS centric way to use containers. You don't need to adopt Docker to get the benefits of containers. You adopt Docker to get the benefits of Docker. A lot of the messaging and marketing tends to conflate the two erroneously.

    Docker runs containers slightly differently than LXC. Docker runs the container OS without an init so it can only run a single app, removes storage persistence and builds the container with layers of aufs. This adds a certain amount of complexity to containers.

    Unless you need a container that can only run a single app without storage persistence, there is little need to deal with the additional complexity.

    I see users struggle with single app containers, non persistent data and layers when they simply need a container as a lightweight VM

    LXC containers are simpler to use and provide advanced capabilities. The fledgling Flockport App store provides easy to use apps and web stacks that can be deployed in minutes on any server, a multi distribution LXC installer and a lightweight VM that let's you try LXC and the app store in minutes.

    OS containers offer near seamless migration from your VM workloads. Whatever works for your VM deployments will work for OS containers. The ability to gain the advantages of container technology without needing to re engineer how you deploy applications is an incredible value proposition. App containers are a constrained OS environment and a lot of things don't work as you would expect and you need to expend extra effort to make them work the 'docker way'..

    Scalability

    The PAAS centric view of things has a disproportionate focus pure on app instances leaving app and user data to the end user to manage. The paradox is scalability is completely app and user data dependent. Simply building app instances does not address scalability.

    There are 3 problems of scale; storage, networking and app and user data, and these are large problems with decades of domain solutions behind them. Containers are building blocks but scaling requires a different set of technologies and skillsets independent of containers, VMs or bare metal servers.
